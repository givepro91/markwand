# Markwand UX Stability Policy

Toss 스타일에서 차용한 운영 원칙이다. 공식 Toss 문서를 복제하지 않고, Markwand의 뷰어 제품에 맞게 신뢰와 안정감을 검증 가능한 기준으로 바꾼다.

## 1. 확신 없는 경고는 기본 노출하지 않는다

- 자동 판단이 불완전하면 위험 UI가 아니라 보조 정보로 둔다.
- 사용자에게 “깨진 것처럼” 보이는 배지는 재현 가능한 근거가 있을 때만 상시 노출한다.
- 확인이 필요한 기능은 클릭 후 상세에서만 보여주고, 기본 화면은 읽기 흐름을 방해하지 않는다.

## 2. 첫 화면의 선택지는 적게 둔다

- 읽기 화면의 주 작업은 위치 열기, 앱으로 열기, 검색, 목차다.
- 복사처럼 세부 선택지가 많은 보조 행동은 하나의 메뉴로 묶는다.
- 아직 완성된 편집 경험이 없는 생성/삭제/이름 변경 기능은 노출하지 않는다.

## 3. 뷰어는 가독성이 기능이다

- 본문 폭은 긴 줄을 만들지 않도록 제한한다.
- 본문은 16px 이상, line-height 1.7 이상을 유지한다.
- 본문/보조 텍스트/링크 색은 WCAG AA 대비를 통과해야 한다.
- 색과 폰트 변경은 테스트 가능한 기준과 함께 적용한다.

## 4. 실패는 조용히 숨기지 않는다

- 사용자가 누른 행동이 실패하면 짧고 구체적인 이유를 toast로 보여준다.
- 실패할 수 있는 외부 앱 실행, 클립보드, 파일 시스템 작업은 테스트에서 성공/실패 경로를 분리한다.

## 5. 기능보다 신뢰를 먼저 출시한다

- 검증되지 않은 기능은 보여주지 않는다.
- UI가 “있다”면 사용자는 동작한다고 믿는다. 구현과 검증이 부족한 기능은 숨기고, 나중에 완성된 흐름으로 다시 낸다.
